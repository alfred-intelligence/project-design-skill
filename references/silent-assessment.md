# Silent Assessment — The A → B Handoff

Operationalization questions (branch strategy, release cadence, distribution, need for
CoC/SECURITY) are *not* asked explicitly in Phase A. They are assessed silently from the
brainstorming output, logged in `01-whitepaper.md` under "Assumptions for Phase B", and
presented at the Phase B handoff as recommendations with motivation.

**License is no longer in this list.** As of the current version of this skill, license
is an *explicit* Phase A decision — see SKILL.md § "Explicit decisions in Phase A". The
silent-assessment pattern below applies to everything *except* license.

---

## Assessment axes and their signals

### Strictness level (`solo` / `solo+contrib` / `team`)
**Signals from Phase A:**
- Operator mentions building alone, no other contributors → `solo`
- Private repo, but "maybe open source later" → `solo` with option to move
- Operator mentions the product is public from day one and they accept PRs → `solo+contrib`
- Operator mentions a team, employees, or several regular developers → `team`

### Open source vs private repo
**Signals:**
- Learning project, internal tool, hobby without a distribution plan → usually private
- Library/CLI, platform for other developers, or product with a community angle →
  usually public
- Commercial SaaS → private (source) plus public product website is the common pattern

### Release cadence
**Signals:**
- Operator mentions Conventional Commits or release-please → continuous via release-please
- Large releases with milestones → calendar-based or manual SemVer
- Learning project without distribution on day one → ad-hoc or continuous (for routine)

### Distribution
**Signals:**
- CLI/library → cargo install / npm / pip / go install + binaries from Releases
- SaaS → container + deployment pipeline
- Desktop app → cargo-dist or equivalent for platform binaries
- Internal tool → often an open question, documented as TBD

### CoC and SECURITY
**Signals:**
- Private + solo → both skipped
- Public → both generated (Contributor Covenant + standard template)
- Team (even private) → CoC generated, SECURITY scales with data sensitivity

### Documentation license
**Default mapping from source license:**

| Source license | Doc license (default) |
|---|---|
| MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause, ISC | CC-BY 4.0 |
| GPL-2.0, GPL-3.0, AGPL-3.0, MPL-2.0, LGPL-2.1, LGPL-3.0 | CC-BY-SA 4.0 |
| BSL, SSPL | "All Rights Reserved" (proprietary docs) |
| "All Rights Reserved" / proprietary source | "All Rights Reserved" |

**Rationale:** CC licenses are written for prose and illustrations; source-code
licenses are written for code. Mixing them keeps the legal framing clean.
Share-alike (CC-BY-SA) matches the copyleft principle of GPL/MPL/AGPL; CC-BY matches
the permissive principle of MIT/Apache.

**Scope of "documentation":**

- *Covered* by the documentation license: `README.md`, `docs/`, `{project}-design/`,
  and other top-level `.md` files in the repo.
- *Not covered* (each has its own license): `CONTRIBUTING.md` (project's choice),
  `CODE_OF_CONDUCT.md` (Contributor Covenant is CC-BY 4.0 itself), `SECURITY.md`
  (project's choice), inline source-code comments (covered by the source license).

**Additional signal:** if the operator's other projects use a consistent
documentation license, follow that pattern instead of the table above. Consistency
across an operator's portfolio is usually preferable to a per-project optimum.

**Reuse-style SPDX headers** in every doc file are not generated by default — that
is overkill for most projects. The operator can adopt REUSE later if needed.

### Branch protection strictness
Maps 1:1 to the strictness level. `solo` → 0 reviewers + CI green. `solo+contrib` →
1 reviewer + CI green. `team` → 2 reviewers + CodeOwners + dismiss stale.

## Preserving assumptions

Assumptions are logged in `01-whitepaper.md` under a short "Assumptions for Phase B"
section before Phase A is delivered. Format:

```markdown
## Assumptions for Phase B

The following operationalization assumptions are carried into Phase B for confirmation:
- Strictness level: `solo` (motivation: no external contributors mentioned)
- Repo: private (motivation: learning project without distribution plan)
- Release cadence: continuous via release-please (motivation: operator mentions this explicitly)
- Distribution: TBD (motivation: left open in Phase A; decided in Phase B or later milestone)
- CoC/SECURITY: skipped (motivation: private + solo)
- Documentation license: CC-BY 4.0 (motivation: source license is MIT — permissive)
```

(Note: license is **not** an assumption — it was decided explicitly in Phase A and is
recorded as a fact, not a recommendation.)

This is not cement — the list is a proposal to the operator at Phase B start, and is
preserved so information is not lost if the session changes.

---

## Presentation at Phase B start

Assumptions are presented as concrete recommendations with motivation, not as open
questions. Format per item:

> **Assumption:** [concrete choice]
> **Motivation:** [why this outcome, based on Phase A]
> **Approve or adjust?**

The operator can approve in bulk or adjust individual items. Once all items are
confirmed, the agent moves on to revising 03/04 (if necessary) and then generating
05–07 + bootstrap.

---

## What does *not* belong as a silent assumption

Anything that touches the product itself — features, scope, target audience,
architecture, data model — is Phase A work and must be explored explicitly during
brainstorming. Silent assumption applies only to operationalization questions that don't
affect the product itself, only how it is run.

If an operationalization question turns out to affect the product (for example, a
distribution choice that affects architecture) — raise it explicitly in Phase A instead
of assessing silently.

Anything in the "must not assume implicit OK" categories from SKILL.md § Open questions
tracking is also off-limits to silent assessment by definition. Re-read that section
before deciding to defer a question.

## Phase C — mode changes mid-project

Implementation mode is decided explicitly in Phase A, but the project may need to
switch modes later. The agent should detect signals and raise the question
explicitly — never switch silently.

**Signals for `iterative` → `spec-first`:**

- Operator starts insisting that design docs match the code exactly before merge
- release-please is added with an expectation of rich, accurate changelogs
- Operator wants to run the implementer autonomously overnight or in batches
- Commit-history quality complaints ("the changelog is useless")

**Signals for `spec-first` → `iterative`:**

- Operator wants to revise 03 or 04 mid-build based on what implementation revealed
- Pre-approved steps repeatedly fail to match reality on first execution
- Operator wants to be more hands-on and review as work unfolds
- The project pivots scope and a fresh detailed spec would cost more than just letting
  design evolve

**Cost of switching:**

- `iterative` → `spec-first` is **expensive**. Requires revising 03 to step-by-step
  detail, revising 04 with granularity + narration rules, switching merge strategy in
  branch-protection.json, and accepting that release-please-relevant history before
  the switch may be lossy.
- `spec-first` → `iterative` is **cheap**. Mostly relaxing rules — drop the granularity
  mandate, allow squash-merge, drop the narration requirement. The detailed 03 and 04
  can stay as-is until they're updated naturally.

The agent presents the mode change as a recommendation with motivation and the cost
delta, lists affected documents and artifacts, and waits for approval before any
change is written. As with strictness-level changes (see `agent-loop-templates.md`
§ Update mode), recommend in both directions — loop creep applies to mode just as it
applies to strictness.
