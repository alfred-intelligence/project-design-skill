## License — guidance

| License | When to choose |
|---------|----------------|
| MIT | Maximum permissivity; short and well known |
| Apache-2.0 | Permissive + explicit patent grant; default for libraries |
| MPL-2.0 | Mild copyleft per file; compromise between open and protected |
| AGPL-3.0 | Strong copyleft including network use; SaaS protection |
| BSL / SSPL | Source-available but not OSI-approved; commercial protection |
| "All Rights Reserved" | Private repo, no external license |

**Rules of thumb:**
- Library/CLI for broad use → MIT or Apache-2.0.
- Backend/SaaS one wants to protect → AGPL-3.0 or BSL.
- Private company code → no open-source license; mark as proprietary.
- Compatibility with existing dependencies must be verified.

(Reminder: license is an explicit decision in Phase A — see SKILL.md § Explicit
decisions in Phase A. The handbook records the choice and motivation; it does not
re-decide it.)

### Documentation license

A separate license for documentation is generated by default. CC licenses are
written for prose and illustrations; source-code licenses are written for code.
Mixing them keeps the legal framing clean.

| Source license | Doc license (default) |
|---|---|
| MIT, Apache-2.0, BSD-*, ISC | CC-BY 4.0 |
| GPL-*, AGPL-3.0, MPL-2.0, LGPL-* | CC-BY-SA 4.0 |
| BSL, SSPL, proprietary | "All Rights Reserved" (proprietary docs) |

**Scope of "documentation"** (covered by the doc license):
- `README.md`
- `docs/` folder
- `{project}-design/` folder
- Other top-level `.md` files

**Not covered** (each has its own license):
- `CONTRIBUTING.md` — project's choice
- `CODE_OF_CONDUCT.md` — Contributor Covenant is CC-BY 4.0 itself
- `SECURITY.md` — project's choice
- Inline source-code comments — covered by the source license

LICENSE-docs file: see `bootstrap/LICENSE-docs`.

## Branch strategy — three standard models

### Trunk-based (recommended for solo / small teams)

```markdown
**Model:** Trunk-based development.

- `main` is always deployable.
- Short feature branches (a few days at most) named `<type>/<scope>` (e.g.
  `feat/auth-jwt`).
- Branches are merged via PR after CI is green.
- No long-lived branches besides `main`.
- Tags mark releases (`v1.0.0`).
```

### GitHub Flow (recommended for solo+contrib)

```markdown
**Model:** GitHub Flow.

- `main` is always deployable.
- All work happens in feature branches from `main`.
- A PR against `main` requires CI green + at least 1 review.
- Merge → deploy (or close-to-deploy).
```

### GitFlow (only when needed; heavy releases)

```markdown
**Model:** GitFlow.

- `main` = latest production version (tagged releases).
- `develop` = integration branch.
- `feature/*`, `release/*`, `hotfix/*` per classic GitFlow.

Use only if release cycles are long and several versions are supported in parallel.
GitFlow is overkill for most modern projects.
```

## Commit conventions — Conventional Commits

```markdown
**Format:** `<type>(<scope>): <subject>`

**Types:**
| Type | Use |
|------|-----|
| feat | New feature (user-visible) |
| fix | Bug fix |
| docs | Documentation only |
| style | Formatting, no code changes |
| refactor | Code change without functional change |
| test | Add or change tests |
| chore | Maintenance (deps, build, etc.) |
| ci | CI/CD config |
| build | Build system, external dependencies |
| perf | Performance improvement |

**Subject rules:**
- Imperative form ("add", not "added").
- ≤ 72 characters.
- No trailing period.

**Body (optional but encouraged):**
- Explain *why*, not *what* (the *what* is visible in the diff).
- Wrap lines at ~72 characters.

**Granularity:**
- The reason is **traceability first** — a legible per-step history, and in `spec-first`
  the ability to monitor an autonomous run commit by commit. A changelog tool like
  release-please is *one consumer* of that granularity when present, not the reason for it;
  projects with no such tool (deploy-as-release, apply-as-release) keep the same
  granularity for traceability alone.
- One commit per logical change, never batch unrelated changes into a single commit.
- In `iterative` mode this is recommended for a readable git log, but squash-merge at
  PR time is acceptable because each PR is a single logical unit.
- In `spec-first` mode this is **non-negotiable** and combined with rebase-merge.
  release-please reads each commit since the last release tag; batched commits collapse
  changelog entries and break version-bump detection. See § Release process.

**Breaking changes:**
- `feat!: change API X` OR footer `BREAKING CHANGE: <explanation>`.

**Example:**
```
feat(auth): add JWT refresh-token endpoint

Previously the full login flow was repeated at token expiration. The new
endpoint /auth/refresh returns a new access token without requiring user
interaction. Refresh tokens are rotated on each use.

Closes #142
```
```

## PR process per level

### Solo

```markdown
**PR size:** Guideline < 400 lines. Larger PRs self-motivated in the description.

**PR description should have:**
- A brief what-section
- An answer to why

**Requirements for merge:**
- CI green
- No unresolved conflicts

**Reviewers:** 0 (self-review)
**Merge strategy:** Squash in `iterative` mode; rebase in `spec-first` mode (preserves granular commits for release-please)
**Auto-merge:** Enabled after CI green
```

### Solo+contrib

```markdown
**PR size:** Guideline < 400 lines.

**PR description should have:**
- What-section
- Why (link to issue if applicable)
- How it was tested
- Migration or config changes

**Requirements for merge:**
- CI green
- 1 review
- Conventional Commits-formatted PR title
- No unresolved discussions

**Reviewers:** 1 (CODEOWNERS = the project owner)
**Merge strategy:** Squash in `iterative` mode; rebase in `spec-first` mode (preserves granular commits for release-please)
**Auto-merge:** Enabled after all checks green
```

### Team

```markdown
**PR size:** Guideline < 400 lines. Larger PRs split or motivated.

**PR description should have:**
- What-section
- Why with issue link
- How it was tested (including manual steps)
- Migration or config changes
- Risks and rollback plan

**Requirements for merge:**
- CI green
- 2 reviews
- Required reviewers from CODEOWNERS for affected files
- Dismiss stale reviews when a new commit is pushed
- No unresolved discussions
- Linear history (rebase or squash, no merge commits)

**Reviewers:** 2
**Merge strategy:** Squash (default in `iterative` mode); rebase in `spec-first` mode (preserves granular commits for release-please)
**Auto-merge:** Manual after approval
```

## Code review checklist (generic)

```markdown
**Review the following at code review:**

### Function
- [ ] Does the PR solve what it describes?
- [ ] Tests cover both happy paths and failure cases.
- [ ] Edge cases handled.

### Code
- [ ] Clear names for variables, functions, types.
- [ ] No unnecessary comments ("// increment i by 1").
- [ ] No magic numbers or hardcoded values.
- [ ] Error handling consistent (returns / errors / exceptions).

### Security
- [ ] No secrets in code or commit history.
- [ ] Input validated before use.
- [ ] Auth/authz at the right level.

### Maintenance
- [ ] Backwards-compatible, or migration clearly documented.
- [ ] Dependencies motivated (why this particular package?).
- [ ] Logging/observability added where needed.
```

## Release process — patterns by delivery model

The pattern follows the delivery model (see SKILL.md § Delivery model). Artifact projects
tag versions — Continuous or Manual SemVer below. Deployment and infra projects ship no
versioned artifact — Deploy-as-release or Apply-as-release below.

### Continuous (release-please)

```markdown
**Tool:** [release-please](https://github.com/googleapis/release-please).

**Flow:**
1. PRs are merged to `main` with Conventional Commits.
2. The release-please bot maintains an ongoing "Release PR" that accumulates the
   changelog.
3. When the operator merges the Release PR: a tag is created, a GitHub Release is
   published, a version bump is pushed.
4. `release.yml` is triggered by the tag and builds artifacts.

**Version bumps (auto):**
- `feat` → MINOR
- `fix`, `perf` → PATCH
- `feat!` or `BREAKING CHANGE` → MAJOR

**Commit granularity is critical.** release-please reads each commit since the last
release tag. Batched or vague commits (`chore: misc updates`) produce empty changelog
entries and miss version bumps. The implementing agent commits per logical change.
Merge-strategy interaction:

- With `squash-merge`, each PR's commits collapse into a single commit message taken
  from the PR title. This works only if **every PR is a single logical change**
  (`iterative` mode default).
- With `rebase-merge`, the granular commits on the feature branch are preserved into
  `main`. Required for `spec-first` mode, where PRs may contain multiple pre-approved
  steps each of which deserves its own changelog entry.

Choose merge strategy accordingly when release-please is in use.
```

### Manual SemVer

```markdown
**Tool:** Manual tag + GitHub Release UI or `gh release create`.

**Flow:**
1. Decide a new version based on Conventional Commits since the previous tag.
2. Update `CHANGELOG.md` manually.
3. Commit the changelog: `chore(release): vX.Y.Z`.
4. Tag: `git tag -a vX.Y.Z -m "Release X.Y.Z"` + `git push --tags`.
5. `release.yml` builds and publishes.
6. Create a GitHub Release with notes from the changelog.
```

### Deploy-as-release (deployed sites and services)

```markdown
**For:** delivery model = deployment — a static site, CMS front, or continuously deployed
service, where there is no versioned artifact to tag.

**Flow:**
1. A change merges, or content is published from the CMS.
2. The build/render runs and deploys to the target (Pages, the cluster, the host).
3. A post-deploy gate — a smoke or health check — decides whether the deploy is accepted.
4. "Release" = a deploy that passes the gate. No SemVer tag is required; optionally tag a
   notable state (`milestone/go-live`).

**Versioning:** usually none — the live deployment is the version. Rollback is "promote
the last known-good deployment", not "check out a tag".

**The smoke gate is the real signal.** A zero exit from the deploy step is not success;
the post-deploy check that the site or service actually serves is. Define it in 06.
```

### Apply-as-release (infrastructure / config / IaC)

```markdown
**For:** delivery model = infra/config — Terraform, Pulumi, cluster manifests, runbooks.

**Flow:**
1. A change merges.
2. `plan` (or an equivalent dry-run) runs and is reviewed.
3. `apply` runs against the environment.
4. A post-apply check — drift or health — confirms the desired state.
5. "Release" = an applied change that verifies. Tag per environment if useful
   (`prod-2026-06-12`), not SemVer.

**Versioning:** environment-tagged or none — the applied state is the version. Rollback is
re-applying the last known-good revision.
```

## Maintenance policies

```markdown
## Maintenance

### Dependency updates

- **Tool:** Dependabot (config: `bootstrap/.github/dependabot.yml`).
- **Frequency:** Weekly (security: daily).
- **Auto-merge:** Patch updates with green CI.
- **Major updates:** Manual review, run in a separate branch.

### Security patching

| Severity | SLA for fix |
|----------|-------------|
| Critical | 7 days |
| High | 30 days |
| Medium | 90 days |
| Low | Best effort |

Sources: GitHub Security Advisories, CVE feed for relevant dependencies.

### Deprecation

- Mark deprecation in code (`@deprecated` / comment / lint warning).
- Document in `CHANGELOG.md` and `MIGRATIONS.md`.
- Keep for ≥ 1 MAJOR version before removal.

### EOL policy

- At minimum the latest two MAJOR versions are actively supported with security
  patches.
- EOL is announced at least 6 months in advance via `CHANGELOG.md` and GitHub
  Discussions.
```
